# LimitLens Architecture

## Opening

LimitLens is a local-first observability system for AI usage pressure that is designed to prevent workflow disruption before provider limits are hit. The product is intentionally split into two user surfaces, a terminal snapshot interface and an always-on menu bar interface, but both are projections of one shared semantic core. That shared core reads local provider artifacts, normalizes heterogeneous evidence into one coherent model, evaluates threshold policy, and emits user-facing severity signals.

The architecture is built for incremental growth in two dimensions. The first dimension is provider growth, where built-in providers and runtime-configured providers can coexist without forcing schema rewrites. The second dimension is distribution growth, where the same runtime can be consumed through local source builds, signed application installs, DMG delivery, and Homebrew packaging. Milestone v0.5 extends this second dimension by adding onboarding behavior, remediation-aware error UX, release artifact automation, and public-facing project scaffolding.

## Ontology

The central entity is `ProviderSnapshot` in `Sources/LimitLensCore/Models.swift`. A provider snapshot represents one provider at one capture instant and carries identity, confidence, pressure metrics, historical rate-limit evidence, source provenance, structured errors, and remediation guidance. The remediation field is intentionally first-class because error states are treated as actionable product states rather than silent failure conditions.

Provider identity is represented by `ProviderName` and `ProviderDescriptor`, which allow stable built-in ordering while preserving extensibility for custom provider slugs. Multi-provider state is represented by `GlobalSnapshot`, which both the CLI and menu bar consume. Threshold behavior is represented by `LimitLensSettings`, `ThresholdEvent`, and `ThresholdRuntimeState`. Runtime state now also carries onboarding lifecycle markers, including one-time CLI welcome display and menu welcome visibility timing, so onboarding behavior can be deterministic across restarts.

Onboarding semantics are represented by `ProviderDetectionStatus` in `Sources/LimitLensCore/ProviderDetection.swift`, which evaluates whether each provider is detected based on concrete evidence in snapshots. Severity semantics remain represented by `SeverityLevel` and are evaluated through `SeverityEvaluator`, which ties warning and critical meanings to user thresholds instead of fixed constants.

## Geography

At repository root, `Package.swift` defines a multi-target Swift package with `LimitLensCore` as the shared library, `LimitLensMenuBarSupport` as the support library for launch and notification policy, and executable targets for the CLI, menu bar app, and two test runners. The root `README.md` is now distribution-oriented and includes visual evidence assets from `docs/images`. The root `CONTRIBUTING.md` defines contributor workflow and review expectations, while `LICENSE.md` remains MIT.

The core domain and policy live under `Sources/LimitLensCore`. The file `Models.swift` defines shared models and persisted state. The file `ProviderAdapters.swift` implements built-in and runtime adapter collection, including path-health diagnostics and remediation hints for error cases. The file `ProviderDetection.swift` maps normalized snapshots to onboarding detection summaries. The file `ProviderRegistry.swift` centralizes adapter registration and extension policy. The file `SnapshotService.swift` aggregates all adapters and normalizes fallback remediation when adapters omit it. The files `ProviderParsing.swift`, `FileSystemSupport.swift`, and `ProcessSupport.swift` isolate parsing, filesystem, and subprocess responsibilities. The files `ThresholdEngine.swift`, `SeverityEvaluator.swift`, and `SnapshotFormatter.swift` define alert semantics and terminal/status formatting.

The CLI entrypoint is `Sources/LimitLensCLI/main.swift`, which parses flags, loads settings and runtime state, applies path overrides, renders first-run onboarding once, and emits either human or JSON snapshots. The menu app entrypoint is `Sources/LimitLensMenuBar/main.swift`, which owns status item lifecycle, polling cadence, threshold notifications, launch-at-login integration, onboarding panel visibility, and provider-specific fix actions. The settings window implementation is `Sources/LimitLensMenuBar/SettingsWindowController.swift`, which now supports focused field routing so fix actions can land directly on the relevant source path control.

Menu support infrastructure lives in `Sources/LimitLensMenuBarSupport`, where `NotificationCoordinator.swift` manages authorization-aware delivery policy and `LaunchAtLoginManager.swift` manages LaunchAgent lifecycle execution. Validation harnesses live in `Sources/LimitLensCoreTestsRunner` and `Sources/LimitLensMenuBarTestsRunner`, with fixture data under `Sources/LimitLensCoreTestsRunner/Fixtures`.

Distribution tooling lives in `scripts`. The file `scripts/install.sh` builds universal binaries, installs local artifacts, and supports explicit signing and notarization controls with an unsigned fallback. The file `scripts/build-dmg.sh` assembles versioned DMG artifacts with `LimitLens.app` and `Applications` shortcut layout. The file `scripts/build-release-assets.sh` composes release-grade CLI and DMG outputs, while `scripts/create-release.sh` publishes GitHub releases with notes and attached artifacts. Homebrew publication helpers live in `scripts/update-homebrew-formulas.sh` and `scripts/publish-homebrew-tap.sh`, and tap assets live under `packaging/homebrew` with Formula and Cask definitions.

Project automation lives under `.github/workflows`, where `ci.yml` validates build and test harnesses and `release.yml` builds artifacts on tag pushes and publishes release entries with attached binaries and changelog files. Public release notes live in `docs/releases`, distribution instructions live in `docs/distribution.md`, and provider extension documentation lives in `docs/adding-providers.md`.

## Flow

In CLI flow, execution begins in `Sources/LimitLensCLI/main.swift`, where settings and runtime state are loaded with diagnostics and command-line overrides are applied. The CLI then calls `SnapshotService`, which collects provider snapshots through built-in and runtime adapters, injects generic remediation when needed, and returns an ordered `GlobalSnapshot`. On first run, the CLI calls `ProviderDetectionEvaluator` and prints a boxed onboarding summary before rendering the snapshot. The onboarding marker is then persisted in runtime state so this introduction appears only once.

In menu flow, the app launches as an accessory process, restores runtime state, marks first-seen onboarding time if needed, and starts periodic refresh. Refresh work is split into detached utility-priority collection and main-actor UI application so menu rendering remains responsive. Each cycle reloads settings, collects a snapshot, evaluates threshold crossing policy, persists runtime state, optionally dispatches notifications, and rebuilds the menu. The menu includes a welcome block while onboarding is active, and that block is dismissed either when settings are first saved or after forty-eight hours from first display.

When provider errors occur, adapters emit structured errors and remediation text into `ProviderSnapshot`. The CLI renders these as explicit `error:` and `Try:` lines. The menu renders source issues inline and exposes a `⚠ Fix` action for built-in providers. That action opens the settings window with focus routed to the relevant path field, reducing remediation from a multi-step search into a direct adjustment.

In distribution flow, release artifacts are generated by `scripts/build-release-assets.sh`, which emits a universal CLI tarball and a versioned DMG. `scripts/build-dmg.sh` assembles app bundle metadata, optional signing, optional notarization, and DMG layout packaging. Release publication then uses `scripts/create-release.sh` locally or `.github/workflows/release.yml` in CI to publish tag-based releases with attached binaries and sectioned changelog notes.

## Philosophy

LimitLens favors truthful confidence over performative certainty, and that philosophy now extends to recovery UX. Unknown provider state is treated as a surfaced, explainable state with direct remediation, not a hidden internal condition. The architecture also favors one semantic center with many surfaces, so onboarding detection, severity policy, and remediation guidance are computed once and reused rather than reinterpreted per interface.

The system intentionally balances extensibility and safety. Runtime provider extension is supported to keep adoption fast, but command execution remains opt-in and path-validated to reduce accidental misuse. Distribution automation follows the same principle: scripts support full signing and notarization when credentials exist, but also support unsigned fallbacks with explicit labeling so development and public release paths can coexist without ambiguity.
